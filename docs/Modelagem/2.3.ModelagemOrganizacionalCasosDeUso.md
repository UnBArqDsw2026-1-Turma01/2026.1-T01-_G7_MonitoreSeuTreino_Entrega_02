# 2.3. Modelagem Organizacional: Casos de Uso

## 1. Metodologia

Optou-se pelo **Diagrama de Casos de Uso** para mapear os requisitos funcionais de forma clara e orientada ao valor entregue ao usuário, servindo como uma ponte de comunicação entre a visão de negócios e a modelagem técnica (estática e dinâmica). O artefato visual e suas especificações foram estruturados utilizando a ferramenta **PlantUML**, garantindo padronização e rastreabilidade via versionamento de código.

## 2. Diagrama de Caso de Uso: UC01 - Registrar Sessão de Treino

![](../anexos/img/ModelagemCasosDeUso/Diagrama-de-caso-de-uso.png)

### 2.1. Breve Descrição

Este caso de uso descreve os passos para que um usuário registre os dados e métricas de uma sessão de treinamento realizada no **Sistema de Monitoramento de Treino**. O processo abrange desde a identificação da rotina do usuário até a validação das regras de negócio e a persistência dos dados.

### 2.2. Atores

- **Ator Principal:** `Usuário` (Um praticante que interage com o sistema para registrar seus treinos).
- **Multiplicidade:** 1 Usuário pode iniciar de 0 a múltiplas instâncias deste caso de uso (`1 -- 0..*`).

### 2.3. Relacionamentos (Includes e Extends)

O diagrama mapeia módulos reutilizáveis e fluxos alternativos/exceções através de relacionamentos de inclusão e extensão:

- **Inclusões (`<<include>>`):** Etapas obrigatórias que sempre ocorrem durante o fluxo base.
  - **Autenticar Usuário:** O sistema deve garantir que o usuário está autenticado antes de prosseguir.
  - **Identificar Rotina Ativa:** O sistema verifica qual a rotina de treinos atual do usuário.
  - **Validar Regras de Negócio:** O sistema obrigatoriamente checa as métricas e dados inseridos.
  - **Persistir Sessão Concluída:** O sistema salva os dados no banco de dados ao final do processo.
- **Extensões (`<<extend>>`):** Etapas opcionais ou de desvio que ocorrem apenas sob condições específicas.
  - **Registrar Sessão Avulsa:** Estende o UC01.
    - _Ponto de Extensão:_ Início do Registro.
    - _Condição:_ Ausência de rotina ativa para o usuário.
  - **Exibir Alerta de Erro:** Estende o caso de uso incluído “Validar Regras de Negócio”.
    - _Condição:_ O sistema identifica dados inconsistentes ou ausentes durante a validação.

### 2.4. Pré-condições

- O sistema deve estar operacional e acessível via interface.
- O usuário deve possuir um cadastro prévio válido para ser autenticado com sucesso pelo fluxo `UC_Autenticar`.

---

## 3. Fluxo de Eventos

### 3.1. Fluxo Principal

1. O `Usuário` solicita o registro de uma nova sessão de treino.
2. O sistema executa o caso de uso `Autenticar Usuário` (`<<include>>`).
3. O sistema executa o caso de uso `Identificar Rotina Ativa` (`<<include>>`), carregando os exercícios planejados.
4. O usuário insere as métricas de execução (ex: carga, séries, repetições) e submete o formulário.
5. O sistema executa o caso de uso `Validar Regras de Negócio` (`<<include>>`) para atestar a consistência das informações.
6. Com os dados válidos, o sistema executa o caso de uso `Persistir Sessão Concluída` (`<<include>>`).
7. O sistema informa ao usuário que a sessão foi registrada com sucesso e encerra o caso de uso.

### 3.2. Fluxos Alternativos

- **FA01 - Registrar Sessão Avulsa (`<<extend>>`)**
  1.  No passo 3 do Fluxo Principal, se o sistema identificar a **ausência de rotina ativa**, o fluxo é desviado para este caso de uso alternativo.
  2.  O sistema disponibiliza uma interface limpa onde o usuário pode selecionar manualmente os exercícios desejados do catálogo.
  3.  O usuário realiza a seleção e preenche as métricas de execução.
  4.  O fluxo retorna ao passo 5 do Fluxo Principal.

### 3.3. Fluxos de Exceção

- **FE01 - Exibir Alerta de Erro (`<<extend>>`)**
  1.  No passo 5 do Fluxo Principal (durante o `Validar Regras de Negócio`), se o sistema detectar **dados inconsistentes ou ausentes** (ex: letras em campos numéricos de peso, valores negativos ou campos obrigatórios vazios), este fluxo é acionado.
  2.  O sistema interrompe imediatamente o salvamento e exibe um alerta de erro específico na tela.
  3.  O usuário é solicitado a corrigir as informações destacadas.
  4.  O fluxo retorna ao passo 4 do Fluxo Principal para uma nova submissão.

---

## 4. Pós-condições

- **Cenário de Sucesso:** Uma nova entidade de Sessão e suas subentidades atreladas (exercícios/séries realizadas) são instanciadas e persistidas com sucesso no banco de dados, tornando-se visíveis no histórico do usuário.
- **Cenário de Falha:** Nenhuma alteração é feita no banco de dados. O estado anterior do sistema é mantido e a integridade dos dados é preservada.

---

## 5. Rastreabilidade e Elos com Outros Artefatos

O diagrama de Casos de Uso atua como um elo integrador entre os requisitos em linguagem natural e a arquitetura do sistema:

- **Backlog do Produto:** O `UC01` materializa diretamente as _User Stories_ relacionadas ao registro de treinos e acompanhamento de carga presentes no **Backlog**.
- **Léxico:** Os termos técnicos adotados nos atores e nas etapas (como _Rotina Ativa_, _Sessão Avulsa_ e _Métricas_) estão ancorados na documentação do **Léxico**.
- **Protótipo:** Os fluxos descritos, especialmente a alternância entre interfaces pré-preenchidas e manuais (Sessão Avulsa), refletem a experiência de usuário desenhada nas telas do Protótipo de Alta Fidelidade.
- **Modelagem Dinâmica:** A especificação em passos deste documento dita exatamente a ordem temporal de troca de mensagens ilustrada no **Diagrama de Sequência**.

## 6. Análise Crítica (Senso Crítico)

A utilização estruturada de relacionamentos `<<include>>` e `<<extend>>` provou-se altamente vantajosa para modularizar os requisitos do sistema. Isolar o passo de **“Validar Regras de Negócio”** como um `<<include>>`, por exemplo, garante que esta lógica seja reutilizada posteriormente em cenários de edição de treino.

Adicionalmente, tratar o registro sem rotina como um fluxo de extensão (FA01 - Sessão Avulsa) evitou o engessamento do sistema. Em vez de obrigar todos os usuários a criarem uma rotina prévia rigorosa para usar o aplicativo, o fluxo abraça a flexibilidade, melhorando a adesão ao uso e mantendo o Fluxo Principal coeso.
